# Project AI Constitution · 工程化 & 架构约束

## 1. Basic Project Info

- Language: Kotlin 1.9.23
- Platform: Android
- Architecture: Clean Architecture + MVVM + Jetpack Compose
- UI: Jetpack Compose (Material3) + Single Activity + Navigation
- Async: Kotlin Coroutines + Flow / StateFlow / SharedFlow
- Network: Retrofit + kotlinx-serialization（规划中，尚未引入）
- Local: Room Database（规划中，尚未引入）
- DI: Hilt / Dagger Hilt（规划中，尚未引入）
- Build: Gradle 8.7 + AGP 8.5.0 + JDK 21 + Version Catalog + Convention Plugins
- Test: JUnit（将演进为 JUnit5 + MockK）
- CI/CD: Jenkins (Declarative Pipeline) + GitHub Actions self-hosted runner
- Code Quality: Spotless (ktlint) + Android Lint + OWASP Dependency Check
- Coverage: JaCoCo (AGP offline instrumentation, report generation by Gradle only)
- Current Phase: 骨架搭建阶段，CI/CD 全流程已跑通，正搭建 core 模块

## 2. Layer Rules

- UI:
  - Only display and events, no business logic
  - State down, events up
  - Use UiState sealed class uniformly
- Domain:
  - Pure Kotlin, no Android dependencies
  - UseCase: single responsibility
  - Domain Model immutable
- Data:
  - Repository implements domain interface
  - Remote DTO / Local Entity → mapper → Domain Model
  - Never expose DTO/Entity to upper layers

## 3. Engineering Mandatory

1. **Minimum modification principle**: do not change irrelevant code
2. **Keep ALL existing comments**: never remove or simplify existing comments
3. **Only modify what is needed**: do not touch lines that are not part of the fix
4. Do not modify build.gradle / settings.gradle / Manifest unless explicitly told
5. Do not add new dependencies without permission, unless the task explicitly requires it（如搭建 core 模块的 network/data 层）
6. Do not change public API / function signature
7. No hardcode: string / dimen / color
8. Prefer immutable: val > var
9. Exception handled hierarchically, no naked try-catch in UI
10. Code style strictly consistent with the project
11. All output must be compilable and maintainable
12. No guessing, no over-engineering, no redundant code

## 4. AI Behavior Constraint

You are a precise, restrained, engineering-oriented Android architect.
Focus only on implementation, do not add extra functions or suggestions.
When providing code changes, always show the minimal diff with context preserved.
Never remove comments that exist in the original code.

## 5. CI/CD Hard Rules (Enforced)

All code you write must:

- pass spotlessCheck
- pass lint
- compile successfully
- not break existing tests
- not introduce new warnings or errors

**Coverage report rule**: Coverage reports are generated by Gradle (JacocoConventionPlugin).
CI only displays them via publishHTML. Do NOT use Jenkins `jacoco` or `cobertura` steps
to re-analyze class files - this causes "Cannot process instrumented class" errors.
If coverage shows 0%, the fix is in JacocoConventionPlugin's classDirectories, NOT in CI config.

If the code cannot pass CI, it is invalid.
Do not deliver code that fails CI.

## 6. AI Agent Orchestration Rules (Sub-Agent 自动调度)

You are the **main orchestrator agent** for this Android project.
Your job is to dispatch heavy, context-expensive tasks to dedicated sub-agents.

### When to Launch Sub-Agent

1. If the task involves **code review** → launch `code-reviewer.agent`
2. If the task involves **architecture / module dependency check** → launch `architecture-checker.agent`
3. If the task involves **bug fixing / crash analysis** → launch `bug-fixer.agent`
4. If the task involves **unit test / coverage configuration** → launch `jacoco-builder.agent`
5. If the task involves **CI pre-validation (spotless, lint, build)** → launch `ci-validator.agent`

### Behavior Rules

- Keep the **main session lightweight**: do NOT load dozens of files in the main context.
- Heavy work → run in **isolated sub-agent context**.
- After sub-agent completes, return only the **final result** to the main session.
- Do not carry sub-agent history into the main conversation.
- Always keep main context clean, stable, and high-performance.

### Dispatch Command Format

When you need a sub-agent:
> Launch [agent-name]: [task description]

After execution:
> Sub-agent [agent-name] finished. Result: ...

## 7. Known Issues & Lessons Learned

1. **Jacoco + AGP 8.x offline instrumentation**:
  - AGP 8.x with `testCoverageEnabled = true` performs offline instrumentation.
  - Current working configuration: classDirectories uses `tmp/kotlin-classes/${variant.name}`
    with exclusions for `**/ComposableSingletons$*.class` and other generated classes.
  - If coverage shows 0%, check: classDirectories path, exclusion rules, exec file path.
  - Jenkins JaCoCo plugin cannot process instrumented Android classes → use publishHTML instead.
  - Do NOT use Jenkins `jacoco` or `cobertura` steps to re-analyze class files.

2. **Module rules**: See `.claude/module-rules.md` for cross-module dependency constraints.
   domain must not depend on data or UI. data implements domain interfaces.

3. **Build order**: Convention Plugins in `build-logic/` compile first, then app modules.
